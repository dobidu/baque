<objective>
## Goal
Build `ScatterEngine`, an RT-safe, buffer-domain performance-FX processor that captures the live
stereo mix into a pre-allocated ring buffer and replays it through TR-8-style glitch transforms:
**repeat** (buffer roll), **reverse**, **gate**, and **decimate**. Exposes a `type` (0 = off,
1-10 = glitch presets) and a `depth` [0,1] intensity control. Slice length is beat-synced from BPM.
Standalone only — no FxChain wiring, no p-lock, no APVTS (those are 08-02).

## Purpose
Scatter is the keystone of Phase 8 (Scatter / Perf FX, ESCOPO §4.7). The buffer-domain glitch
primitives (repeat/reverse/gate/decimate/stutter) all share one capture ring + playback-head
manipulation, so they belong in a single processor. Tape-stop (speed ramp) and gater-as-discrete-FX
come later. Mirrors the 07-01 (LoFiProcessor) / 07-03 (GranularProcessor) standalone-first convention:
prove the DSP in isolation with deterministic tests, then wire in the next plan.

<acceptance_criteria>

## AC-1: Bypass passthrough (type 0)
```gherkin
Given a ScatterEngine prepared at 48kHz
When process() is called with type=0 (off)
Then the output buffer is sample-identical to the input (within 1e-6 tolerance)
```

## AC-2: Repeat (buffer roll)
```gherkin
Given a ramp/identifiable signal captured into the ring
When process() runs a repeat-type scatter with a 1/16-beat slice and depth=1.0
Then the output replays the captured slice periodically (slice content recurs at the slice period,
     detectable by comparing samples one slice-length apart)
And the replayed slice is the frozen slice captured in [anchor, anchor+slice_len) where
    anchor = write_pos_ - slice_len at the slice boundary — it does NOT mutate mid-slice even as
    live input continues writing the ring (no self-capture / feedback)   <!-- audit-added M1 -->
```

## AC-3: Reverse
```gherkin
Given a monotonically increasing ramp captured into the ring
When process() runs a reverse-type scatter over a slice with depth=1.0
Then within the active slice the output is time-reversed relative to the captured input
     (output decreases where input increased)
```

## AC-4: Gate
```gherkin
Given a constant non-zero signal
When process() runs a gate-type scatter with depth=1.0
Then the output contains regions of (near-)zero amplitude that are absent from the input
     (gated chops), while non-gated regions preserve the input level
```

## AC-5: Decimate
```gherkin
Given a smooth sine captured into the ring
When process() runs a decimate-type scatter with depth=1.0
Then the output exhibits sample-and-hold stair-stepping (consecutive output samples held equal
     across the decimation stride) not present in the smooth input
```

## AC-6: Depth scales intensity / depth=0 = exact bypass
```gherkin
Given any active type (1-10)
When process() is called with depth=0.0
Then the output is sample-identical to dry input within 1e-6 — the wet/dry math
     out = dry*(1-depth) + wet*depth GUARANTEES exact dry at depth=0, so assert exact, not "near"  <!-- audit-added M1 -->
And increasing depth toward 1.0 monotonically increases the effect's contribution (wet/dry mix)
```

## AC-7: RT-safety (zero allocation in process)
```gherkin
Given a prepared ScatterEngine
When process() is called repeatedly (>= 256 blocks) across all 11 types
Then no heap allocation occurs in process() (all buffers pre-allocated in prepare())
And the call is noexcept and produces no NaN/Inf in the output
```

## AC-8: Beat-synced slice length
```gherkin
Given prepare(sample_rate=48000) and bpm=120
When the slice subdivision is 1/16 note
Then the computed slice length equals samples_per_quarter / 4 = (48000 * 60 / 120) / 4 = 6000 samples
     (verified via an exposed const helper or observable repeat period within tolerance)
```

## AC-9: reset() clears state (no stale replay)   <!-- audit-added SR2 -->
```gherkin
Given a ScatterEngine that has captured signal A and is mid-slice on a repeat-type
When reset() is called, then signal B is fed with a repeat-type scatter
Then the replayed slice contains content from B only — no residue of A's frozen slice
     (ring, write_pos_, read_anchor_, sample_counter_ all zeroed; prevents stale glitch on transport stop/restart)
```

  <files>src/audio/scatter_engine.h, src/audio/scatter_engine.cpp</files>
  <action>
    INVARIANTE DE LEITURA DO RING (M1 — corrige risco de feedback/UAF e ambiguidade de latência):  <!-- audit-added M1 -->
    Toda primitiva lê de uma slice CONGELADA, não do sample que acabou de ser escrito.
      - A fase musical da slice persiste ENTRE blocos via sample_counter_:
        slice_phase = sample_counter_ % slice_len (NÃO reinicie por bloco; a slice tipicamente
        abrange vários blocos de process()).
      - Em cada FRONTEIRA de slice (slice_phase == 0), congele o âncora de leitura:
        read_anchor_ = (write_pos_ - slice_len + ring_size_) % ring_size_  (a slice mais recente,
        que trilha write_pos_ por exatamente slice_len). Mantenha read_anchor_ FIXO durante a slice.
      - Como read_anchor_ trilha write_pos_ por slice_len e ring_size_ >> slice_len, a região lida
        [read_anchor_, read_anchor_+slice_len) e a região escrita durante a reprodução
        [write antigo, write antigo+slice_len) NÃO se sobrepõem → sem auto-captura, sem feedback.
        jassert(slice_len < ring_size_).
      - Todos os índices de ring com wrap circular (% ring_size_); nunca acesse fora de [0, ring_size_).

    COERÊNCIA ESTÉREO (SR1): a fase de slice, read_anchor_, máscara de gate e stride de decimate são  <!-- audit-added SR1 -->
    COMPARTILHADOS entre L e R — só os dados de amostra diferem por canal. Não derive offsets/máscara
    por canal independentemente (rasgaria a imagem estéreo). Espelha o tratamento de canais do LoFi/Granular.

    Implemente as quatro primitivas, todas lendo da slice congelada (não alocam):

    - repeat (buffer roll): reproduz em loop a slice congelada em read_anchor_; re-congela o âncora
      na próxima fronteira de slice. Produz recorrência periódica (AC-2).
    - reverse: lê a slice de trás para frente (índice descendente dentro da slice). Dentro da slice,
      sinal crescente vira decrescente (AC-3).
    - gate: modula amplitude com padrão on/off dentro da slice (ex.: metade on / metade off, ou
      duty derivado de depth). Regiões off → ~0; on → nível de entrada (AC-4).
    - decimate: sample-and-hold com stride (segura 1 sample a cada N; N derivado do tipo/depth),
      gerando degraus (AC-5). Não realocar; apenas segurar valor anterior.

    Dispatch type 1-10 → tabela estática constexpr mapeando cada type para
    (primitiva, subdivisão de slice, intensidade base). Sugestão (documente no header):
      1: repeat 1/8     2: repeat 1/16    3: repeat 1/32    4: reverse 1/8
      5: reverse 1/16   6: gate 1/16      7: gate 1/32      8: decimate (stride leve)
      9: decimate (stride forte)          10: repeat 1/32 + reverse (stutter agressivo)
    Ajuste os números se necessário, mas mantenha cobertura das 4 primitivas e 10 entradas válidas.
    type fora de [0,10] → tratar como 0 (bypass), com jassert.

    depth: além do wet/dry da Task 1, modula a intensidade interna (ex.: duty do gate, stride do
    decimate, nº de repetições) — depth=0 já é bypass via wet/dry, então a intensidade interna pode
    escalar livremente.

    Avoid: qualquer std::vector::resize/push_back, locks, ou IO em process(). Avoid: ler índices de
    ring sem máscara/wrap circular (UB). Avoid: divisão por zero quando bpm<=0 (clamp bpm a >= 1).
  </action>
