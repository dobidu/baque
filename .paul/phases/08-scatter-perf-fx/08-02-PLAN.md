<objective>
## Goal
Integrar o `ScatterEngine` (08-01) no sinal de saída: instanciá-lo em `BaqueProcessor`, prepará-lo,
e chamá-lo em `processBlock` **depois** de `fx_chain_.process()` (opera no mix wet completo, estilo
TR-8 — última etapa de performance). Expor dois parâmetros novos — `scatter_type` (0-10) e
`scatter_depth` (0-1) — via APVTS (default neutro) **e** p-lock (PLockParam 11/12), seguindo o
padrão dos params de Fase 6 (APVTS default + override por step).

## Purpose
08-01 entregou o DSP standalone. Sem wiring, scatter não toca áudio real nem responde a automação/
p-locks. Este plano fecha o caminho: APVTS → snapshot → apply_plock_batch → scatter_.process(), de
modo que scatter automatize por step (como filter/reverb) e seja dirigível ao vivo. Diferente de
lo_fi/granular (p-lock-only, neutros por default), scatter é um FX de performance que precisa de
parâmetros automatizáveis pelo host — daí os params APVTS.

## Output
- `fx_params.h` — campos scatter_type + scatter_depth (default neutro: type 0 = off)
- `plock_pattern.h` — PLockParam::scatter_type=11, scatter_depth=12, count=13
- `plugin_processor.{h,cpp}` — membro ScatterEngine, prepare/reset, APVTS params, snapshot,
  apply_plock_batch cases, scatter_.process() pós-FxChain
- `tests/test_scatter_chain.cpp` — wiring + APVTS + p-lock dispatch + regressão de count
- Asserts de k_plock_param_count atualizados 11 → 13 (3 arquivos)
</objective>

## AC-1: scatter_type / scatter_depth como params APVTS
```gherkin
Given um BaqueProcessor instanciado
When consulto apvts_.getRawParameterValue("scatter_type") e ("scatter_depth")
Then ambos existem (não-null), scatter_type default 0.0 (off, range 0-10 passo 1),
     scatter_depth default 0.0 (range 0-1)
```

## AC-2: PLockParam estendido sem quebrar dispatch existente
```gherkin
Given o enum PLockParam
When adiciono scatter_type=11 e scatter_depth=12
Then k_plock_param_count == 13
And os 11 params anteriores mantêm seus índices (0-10 inalterados)
And apply_plock_batch tem case para scatter_type e scatter_depth (sem cair no default)
```

## AC-3: p-lock sobrescreve scatter params por step
```gherkin
Given um PLockBatch com eventos scatter_type=2 e scatter_depth=1.0
When apply_plock_batch roda sobre um FxParams default
Then fx.scatter_type == 2.0 e fx.scatter_depth == 1.0 (override aplicado)
```

## AC-4: ScatterEngine processa pós-FxChain, neutro = sem efeito
```gherkin
Given um buffer conhecido não-silencioso processado por FxChain
When ScatterEngine roda DEPOIS da FxChain (type>=1, depth>0)
Then a saída do par (FxChain → Scatter) diverge da saída só-FxChain (scatter glitcha o sinal já processado)
And com scatter_type=0 (ou depth=0) a saída do par == saída só-FxChain (scatter em bypass)
```

## AC-4b: wiring verificado em processBlock real (não-vacuoso)   <!-- audit-added M1 -->
```gherkin
Given um BaqueProcessor instanciado + prepareToPlay (precedente: test_transport.cpp)
When processBlock roda múltiplos blocos com scatter ativo via APVTS
Then nenhum crash, nenhum NaN/Inf na saída (prova: scatter_ foi preparado E chamado — se
     prepareToPlay esquecer scatter_.prepare, ring_size_=0 → no-op silencioso; o teste de
     ordering AC-4 cobre a semântica pós-chain que o smoke sozinho não captura)
```

## AC-5: RT-safety preservada
```gherkin
Given processBlock com scatter ativo
When o bloco roda
Then nenhuma alocação/lock/IO ocorre no caminho de áudio (scatter_.process é noexcept,
     ring pré-alocado em prepareToPlay; bpm vem de transport_.bpm já lido)
```
