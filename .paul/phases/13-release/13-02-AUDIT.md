# Audit: Plan 13-02 — CPack + release.yml + codesign + v1.0.0 tag

**Verdict: Conditionally acceptable — upgraded. 2 must-have + 2 strongly-recommended applied.**

## Auditor Role
Senior principal engineer + CI/CD reviewer

## Executive Summary

Two workflow-breaking bugs found that would cause the release job to fail silently or loudly on the first tag push. Both are mechanical YAML/cmake errors, not design flaws. Structural plan is sound: CPack component isolation is correct, codesign gating is correct, release job dependency chain is correct. Four gaps patched.

---

## What Is Solid

- CPack component isolation approach: `CPACK_COMPONENTS_ALL = VST3 Standalone AU` + `CPACK_COMPONENT_UNSPECIFIED_HIDDEN TRUE` correctly limits ZIP contents to plugin artifacts. JUCE headers go into the "Unspecified" component; excluding it from `CPACK_COMPONENTS_ALL` prevents inclusion.
- Codesign step gated on `${{ secrets.APPLE_SIGNING_IDENTITY != '' }}` — clean skip when secret absent; macOS ZIP still produced. No hard failure.
- Release job `needs: [build-linux, build-macos, build-windows]` — release only runs after all 3 builds succeed; correct.
- `CPACK_SOURCE_GENERATOR = ""` — no source tarball; correct for binary release.
- `CPACK_STRIP_FILES TRUE` — strips debug symbols from Linux binary; correct for release ZIP.
- `permissions: contents: write` scoped only to release.yml — least privilege; CI remains `contents: read`.
- `concurrency: cancel-in-progress: false` on release job — never cancel a release run; correct.

## Findings

### M1 — Must-Have: `gh release create` step uses `uses:` instead of `run:`

**Issue:** The release job's "Create GitHub Release" step uses `uses: gh release create ...`. In GitHub Actions YAML, `uses:` accepts only `owner/repo@ref` or `./local/action` references. `gh release create` is a CLI command — it MUST use `run:`. The YAML is structurally valid but semantically broken: GitHub Actions would try to resolve `gh release create v1.0.0 ...` as an action reference, fail immediately, and the release would never be created. The tag is pushed, CI runs, 3 ZIPs are built, the release job fails — no GitHub Release created.

**Fix applied to PLAN.md:** Changed `uses: gh release create ...` to `run: |` block with the `gh release create` command inside.

**Risk if skipped:** Release job fails every time; no GitHub Release created; v1.0.0 tag is stranded without downloadable artifacts.

---

### M2 — Must-Have: `CMAKE_SYSTEM_NAME` is "Darwin" on macOS, not "macOS"

**Issue:** `CPACK_PACKAGE_FILE_NAME "BAQUE-${PROJECT_VERSION}-${CMAKE_SYSTEM_NAME}"` produces `BAQUE-1.0.0-Darwin.zip` on macOS. The plan's AC-5 and human-facing documentation say "macOS". "Darwin" is the kernel name — users downloading from the GitHub Release page would see a confusing name inconsistent with the README install instructions.

**Fix applied to PLAN.md:** Replaced single `CPACK_PACKAGE_FILE_NAME` with platform-specific strings using `if(WIN32)/elseif(APPLE)/else()`: Windows, macOS, Linux. Task 3 verify checklist updated to `BAQUE-1.0.0-macOS.zip` (consistent with AC-5).

**Risk if skipped:** Confusing ZIP names on GitHub Release; "BAQUE-1.0.0-Darwin.zip" vs README's "macOS" path instructions causes user support friction.

---

### SR1 — Strongly Recommended: `artifacts/**/*.zip` glob fails without `shopt -s globstar`

**Issue:** `actions/download-artifact@v4` creates subdirectories:
```
artifacts/
  BAQUE-Linux/BAQUE-1.0.0-Linux.zip
  BAQUE-macOS/BAQUE-1.0.0-macOS.zip
  BAQUE-Windows/BAQUE-1.0.0-Windows.zip
```
Bash's `**` glob requires `shopt -s globstar` which is NOT enabled by default in GitHub Actions' bash shell. Without it, `artifacts/**/*.zip` only matches files directly in `artifacts/` (not subdirectories) — which is zero files. The `gh release create` command would succeed but create a GitHub Release with no attached ZIPs.

**Fix applied to PLAN.md:** Replaced `artifacts/**/*.zip` with `$(find artifacts -name "*.zip" | tr '\n' ' ')` in the run block, evaluated before passing to `gh release create`.

**Risk if skipped:** GitHub Release created with 0 attached files; users cannot download anything; v1.0.0 is a broken release.

---

### SR2 — Strongly Recommended: Task 1 local cpack test missing reconfigure step

**Issue:** Task 1 Step 2 runs `cpack -G ZIP --config CPackConfig.cmake` immediately after editing CMakeLists.txt, without first running `cmake -B build`. `CPackConfig.cmake` is generated by cmake during configure, not during build. If the build directory was configured before the CPack block was added, `CPackConfig.cmake` either doesn't exist or lacks the new `CPACK_PACKAGE_FILE_NAME`, `CPACK_COMPONENTS_ALL`, and other settings. The cpack command would either fail ("CPackConfig.cmake not found") or produce an incorrect artifact.

**Fix applied to PLAN.md:** Added `cmake -B /mnt/d/work/baque/build -DCMAKE_BUILD_TYPE=Release` as the first step in the local test sequence, before the cmake build and cpack commands.

**Risk if skipped:** Local cpack test produces wrong ZIP name or fails; verification step misleads about CI behavior.
