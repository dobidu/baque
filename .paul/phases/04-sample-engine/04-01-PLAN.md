<objective>
## Goal
Replace the single global sample with a 16-pad bank (4×4, ESCOPO §4.1): each pad owns its sample buffer and playback params (gain, pan, pitch semitones+cents, reverse). MIDI note routes to pad; voice plays with varispeed (rate change — pitch and duration linked, SP/vinil character) and optional reverse.

## Purpose
First slice of Phase 4 (Sample Engine). Establishes the pad data model every later plan builds on: choke groups and velocity layers (04-02) attach to pads; chop-to-pads (04-03) fills pads with slices; offline stretch (04-04, fork v1) renders into pad buffers. Varispeed here is in-house (cheap rate change per ESCOPO §4.11 realtime path) — no SoundTouch dependency.

## Output
- `src/audio/pad_bank.h/.cpp` — SamplePad + PadBank (16 pads, note→pad lookup)
- `src/audio/sample_voice.h/.cpp` — fractional-position playback (linear interp), rate, reverse, pan
- `src/audio/scheduler.h/.cpp` — routes note-on → pad → voice with pad params
- `src/plugin_processor.h/.cpp` — owns PadBank; existing WAV load targets pad 0
- `tests/test_pad_bank.cpp` — routing, varispeed, reverse, RT-safety tests
</objective>

<acceptance_criteria>

## AC-1: Note routes to pad
```gherkin
Given a PadBank with distinct samples loaded on pad 0 and pad 1
When note-on 36 and note-on 37 arrive in the same block
Then pad 0's sample and pad 1's sample each play (note 36+index convention)
And a note with no mapped pad (or empty pad) triggers nothing, without error
```

## AC-2: Varispeed pitch
```gherkin
Given a pad with pitch set to +12 semitones, 0 cents
When the pad is triggered
Then the sample plays at 2.0× rate (rate = 2^((semitones + cents/100)/12))
And the audible duration halves (pitch and duration linked — varispeed)
And fractional positions are linearly interpolated (no stepping artifacts)
And at any rate (incl. 4.0× / +24 st) playback never reads past the buffer: the voice deactivates when position reaches the last valid frame <!-- audit-added -->
```

## AC-3: Reverse playback
```gherkin
Given a pad with reverse enabled and a known ramp sample
When the pad is triggered
Then output frames are the sample mirrored end-to-start
```

## AC-4: RT-safety preserved
```gherkin
Given samples loaded on multiple pads and a looping pattern
When processBlock runs with sequencer-generated note-ons
Then zero allocations occur on the audio thread (existing RT-safety test pattern)
And 21 existing tests still pass unchanged in behavior
```

## AC-5: Safe sample load — no dangling voice pointers <!-- audit-added -->
```gherkin
Given a voice actively playing pad 0's sample
When a new sample is loaded into pad 0 (message/background thread)
Then all voices are deactivated BEFORE the pad's buffer is mutated
And no voice ever reads a reallocated/freed buffer (no use-after-free)
And the load path is the only writer of pad buffers (single-writer contract)
```

    document this contract in the header (Portuguese comments per project convention).
    Audio thread only reads raw pointers via getReadPointer. No locks, no atomics needed
    in this plan because Phase 2 established load-before-play usage; full lock-free
    swap is deferred (noted in boundaries).

    This is the documented invariant future UI/automation phases must upgrade
    (atomics or command queue) before allowing live edits.

    Avoid: std::map/unordered_map for note lookup (allocation + cache-hostile);
    arithmetic note−base into array is enough.
  </action>
