<objective>
## Goal
Close the MIDI control loop both directions: (1) emit programmed p-lock automation as
MIDI CC on the EXT channel so BAQUE can drive a TR-8/TR-8S's scatter/FX knobs; (2) accept
inbound MIDI CC from the host/hardware and apply it to internal FX params; (3) MIDI learn —
arm a target, capture the next inbound CC, persist the binding across session save/load.

## Purpose
ESCOPO §4.8 "MIDI I/O — CC out, MIDI learn, hybrid mode". 09-01 gave per-lane note routing,
09-02 gave clock master. CC out is what actually moves a TR-8's scatter depth / FX in lock-step
with the pattern (the product's hybrid-hardware promise). MIDI-in + learn make external
controllers drive BAQUE's FX — the inbound half of hybrid mode.

## Output
- New `src/audio/midi_cc.h`: `CcOutRouting` (PLockParam→CC map, outbound) + `CcLearnMap`
  (inbound CC→PLockParam bindings + learn-arm), both POD single-writer (mirror LaneRouting).
- `plugin_processor.{h,cpp}`: members `cc_out_` + `cc_learn_` (public, single-writer);
  CC-out emit in processBlock; inbound-CC parse + apply; learn capture; state persistence.
- `tests/test_midi_cc.cpp`: CC-out emit, inbound apply, learn arm/bind, state round-trip.
</objective>

<acceptance_criteria>

## AC-1: P-lock fires → CC on EXT channel
```gherkin
Given cc_out_.enabled = true and cc_out_ maps PLockParam::scatter_depth → CC 16 on channel 10
  And a pattern step carries an active scatter_depth p-lock = 0.5
When processBlock runs the block containing that step
Then midi_messages output contains a controllerEvent(channel=10, controller=16, value=64)
  (0.5 normalized over scatter_depth range [0,1] → 64), at the step's sample offset
```

## AC-2: CC out disabled / unmapped params emit nothing
```gherkin
Given cc_out_.enabled = false (or the firing param has no CC mapping)
When a p-lock step fires in processBlock
Then no controllerEvent is added to the output for that param
  (note events and clock are unaffected — CC out is additive and opt-in)
```

## AC-3: Inbound CC applies to bound FX param
```gherkin
Given cc_learn_ binds inbound CC 74 on channel 1 → PLockParam::filter_cutoff
When the host delivers a controllerEvent(channel=1, controller=74, value=127) in midi_messages
Then this block's FxParams.filter_cutoff is set to the param max (20000 Hz)
  And precedence holds: APVTS snapshot < inbound CC < per-step p-lock
```

## AC-4: MIDI learn binds the next inbound CC
```gherkin
Given arm_learn(PLockParam::reverb_mix) was called (learn armed, no binding yet)
When the next inbound controllerEvent(channel=3, controller=22, value=anything) arrives
Then cc_learn_ records binding {cc=22, channel=3 → reverb_mix}, learn disarms
  And a subsequent CC 22 / channel 3 drives reverb_mix (AC-3 path)
```

## AC-5: Learn map persists across save/load
```gherkin
Given cc_learn_ holds one or more bindings
When getStateInformation serializes then setStateInformation restores into a fresh processor
Then the bindings are identical; a pre-09-03 state (no cc_learn child) loads to an empty map
  without error (backward-compatible); a state with an out-of-range target/channel/cc loads
  with that binding dropped or clamped (no OOB enum/channel)
```

## AC-6: Armed learn captures only controllers; CC out is additive
```gherkin
Given arm_learn(PLockParam::filter_cutoff) was called (armed)
When the next inbound event is a note-on (not a controller)
Then no binding is recorded and learn stays armed (only a controller event binds)
And given cc_out_.enabled with a mapping, when a p-lock step fires
Then the EXT note events and MIDI-clock bytes already in the output are unchanged in count and
  position (CC out is purely additive — it must not drop, reorder, or displace notes/clock)
```

<boundaries>

## DO NOT CHANGE
- EXT-buffer ordering invariant (09-02): notes → stop-flush → clock → clear+addEvents. CC-out
  inserts after notes/apply_plock_batch and BEFORE stop-flush/clock. Do not reorder clock.
- `PLockParam` enum + `k_plock_param_count` (= 15, FROZEN). CC maps are KEYED by it, never extend it.
- MidiClock, ScatterEngine, GaterProcessor, TapeStopProcessor and their processBlock order.
- lane_routing_ EXT note emit + stop-flush logic (09-01) — CC out is additive, must not alter notes.
- Final `midi_messages.clear(); midi_messages.addEvents(midi_buffer_ext_, ...)` invariant.

## DO NOT CHANGE (audit-added M1)
- The learn threading contract: `learn_arm` MUST be std::atomic<int>; bindings are audio-thread-
  owned during capture. Do not "simplify" to a plain enum or expose a UI-mutable bindings getter.

## SCOPE LIMITS
- Block-granular CC timing only (offset 0); sub-block sample-accurate CC deferred (PLockEvent has
  no sample offset — out of scope for v1).
- Inbound-CC targets cover ALL 15 PLockParams (SR1 — every one maps to a FxParams field); no
  silent no-op targets. (Coverage is intentionally NOT restricted to the APVTS-snapshot subset.)
- No MIDI-thru, no inbound note remap, no NRPN/14-bit CC, no CC smoothing/ramp (raw → param).
- filter_cutoff CC law is LINEAR for v1; log/perceptual CC mapping deferred.
- cc_out_ is single-writer like LaneRouting (Phase 10 UI atomicizes); cc_learn_ uses the atomic-
  arm + audio-owned-bindings contract above (Phase 10 UI snapshots bindings, mirrors pad-params).
</boundaries>
