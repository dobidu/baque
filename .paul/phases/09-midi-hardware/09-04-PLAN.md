<objective>
## Goal
Ship ready-made TR-8 / TR-8S mapping templates (per-lane note map + MIDI channel + CC-out
slots) that, applied in one call, configure `lane_routing_`, `cc_out_`, and a `StepPattern`
so a hybrid pattern drives the hardware. Then close Phase 9 with a DoD integration test
proving internal sample lanes and external MIDI lanes coexist in the same pattern and emit a
correct merged MIDI stream (notes on the right TR-8 notes/channel + 24ppqn clock + p-lock CC),
plus a manual real-TR-8 sign-off.

## Purpose
ESCOPO §4.8: "Templates de mapeamento prontos para TR-8/TR-8S" and the Phase 9 DoD —
"Dirige TR-8 real em sync; lanes internas+externas no mesmo padrão." 09-01/02/03 built the
mechanism (per-lane EXT note routing, clock master, CC out). This plan supplies the *content*
(the actual TR-8 note/channel/CC map) and verifies the whole hybrid path end-to-end. It is the
last plan of Phase 9 and carries the phase-transition commit (09-01..09-04 together, like Phase 8).

## Output
- New `src/audio/hardware_templates.h`: `HardwareTemplate` POD (lane→note map, default channel,
  PLockParam→CC map) + `tr8_template()` + `tr8s_template()` factories + `apply_template(...)`
  free function that writes into `LaneRouting`, `CcOutRouting`, and a `StepPattern`.
- `plugin_processor.{h,cpp}`: thin `apply_hardware_template(const HardwareTemplate&)` convenience
  that applies to `lane_routing_` + `cc_out_` + the active pattern (message-thread, single-writer).
- `tests/test_hardware_templates.cpp`: template content + apply correctness.
- `tests/test_phase9_dod.cpp`: hybrid INT/EXT processBlock integration (the automated half of the DoD).
- Manual real-TR-8 verification (checkpoint) + a documented sign-off procedure/checklist.
</objective>

<acceptance_criteria>

## AC-1: TR-8 template carries the correct note/channel map
```gherkin
Given tr8_template()
When its note map is inspected
Then lanes 0..10 map to the Roland TR-8 GM drum notes on MIDI channel 10:
  BD=36, SD=38, LT=43, MT=47, HT=50, RS=37, HC=39, CH=42, OH=46, CC=49, RC=51
  And every mapped lane defaults to LaneMode::external (or both for lanes flagged hybrid)
  And lanes 11..15 are left as internal (no EXT note assigned)
```

## AC-2: Applying a template configures routing + pattern + CC out in one call
```gherkin
Given a default LaneRouting, CcOutRouting, and StepPattern
When apply_template(tr8_template(), routing, cc_out, pattern) runs
Then routing.mode[lane]/channel[lane] are set per the template for every mapped lane
  And pattern.get_note(lane, step) == the template note for every step of each mapped lane
  And cc_out is configured per the template (channel + the CC slots the template enables)
  And unmapped lanes (11..15) are untouched (mode stays internal)
```

## AC-2b: Template apply preserves pattern programming (non-destructive)   <!-- audit-added SR1 -->
```gherkin
Given a StepPattern with active steps, trig conditions, and notes already programmed on a lane
When apply_template(tr8_template(), routing, cc_out, pattern) runs
Then for each MAPPED lane only the per-step NOTE is rewritten to the template note
  And is_active(lane,step) and get_trig(lane,step) are UNCHANGED for every step
  And unmapped lanes keep their notes AND activations AND trigs entirely
  (routing + cc_out ARE fully reset to the template — this reset is intentional and documented;
   step programming/p-locks are NEVER wiped by applying a hardware template)
```

## AC-3: TR-8S template reuses the TR-8 note map and is selectable independently
```gherkin
Given tr8s_template()
When compared to tr8_template()
Then the core 11-voice note map and channel are identical (TR-8S default GM layout)
  And the template carries a distinct name/id so UI can present both
  And any TR-8S-only CC slots are additive, never altering the shared note map
```

## AC-4: Phase 9 DoD — hybrid pattern emits a correct merged MIDI stream
```gherkin
Given a BaqueProcessor with tr8_template() applied, clock_master_ = true, and a pattern where
  some lanes are EXT (TR-8 notes) and at least one lane is INT (sample), transport playing
When processBlock runs the block(s) covering the active steps
Then midi_messages out contains note-on for each EXT lane on its TR-8 note + channel 10
  And the matching EXT note-OFF appears when the gate closes (no stuck hardware note)   <!-- audit-added SR2 -->
  And contains EXACTLY the 24ppqn clock (0xF8) count expected for the elapsed ppq        <!-- audit-added SR2: exact, not >0 -->
  And INT lanes produce NO MIDI note-on in the output (they trigger internal voices, not MIDI)
  And event ordering is intact (notes → clock; CC additive) — no dropped/reordered events
```

## AC-4b: DoD INT-lane check is non-vacuous (proven active + internal)   <!-- audit-added M1 -->
```gherkin
Given the AC-4 hybrid setup with a pad sample loaded and an INT lane carrying an active step
When the block covering that INT step is processed via BaqueProcessor::processBlock
Then the audio output for that block is NON-ZERO (the internal voice actually fired)
  And NO MIDI note-on for that INT lane's note appears in the output buffer
  (i.e. "no MIDI out" is meaningful BECAUSE the lane provably triggered — not because it was idle)
```

## AC-5: Phase 9 DoD — sustained hybrid run is stable (no dropout, no stuck notes)
```gherkin
Given the AC-4 hybrid setup
When processBlock is driven for 200 consecutive blocks across loop boundaries, then transport stops
Then output is finite every block (no NaN/inf), the per-block MIDI buffer stays consistent
  And on the play→stop edge an All-Notes-Off (CC123) is emitted on each EXT channel (09-01 stop-flush)
  And no assertion fires
```

## AC-6: Real TR-8 / TR-8S hardware sign-off (manual)
```gherkin
Given BAQUE standalone connected to a real TR-8 or TR-8S via MIDI out (template applied, clock master)
When a hybrid pattern plays
Then the hardware triggers the correct instruments in time, follows BAQUE's clock (start/stop/tempo)
  And p-lock CC moves the mapped hardware parameter (or the operator records which CC numbers the
  hardware actually responds to, finalizing the template's CC slots)
```

## DO NOT CHANGE
- EXT-buffer ordering invariant (09-02): notes → stop-flush → clock → clear+addEvents.
- `PLockParam` enum + `k_plock_param_count` (= 15, FROZEN). Templates KEY CC maps by it, never extend.
- MidiClock / ScatterEngine / GaterProcessor / TapeStopProcessor and their processBlock order.
- lane_routing_ EXT note emit + stop-flush (09-01); cc_out_ emit (09-03). Templates only WRITE these
  structs at setup time — they do not change the emit logic.
- midi_cc.h k_param_range / norm / denorm (09-03 SR1). Templates reuse, never duplicate.

## DO NOT CHANGE (audit-added M1 / SR1)
- The DoD test MUST exercise BaqueProcessor::processBlock, never Sequencer::generate directly —
  that is the only path proving lane_routing_/cc_out_/clock are wired together (08-02 M1 lesson).
- apply_template MUST be non-destructive to step programming: set_note only; never set_active /
  set_trig / clear. Do not "simplify" the processor apply to start from a fresh default pattern.

## SCOPE LIMITS
- Templates ship the NOTE map populated (documented Roland GM layout) and CC slots OFF by default;
  confirmed CC numbers are filled at the hardware checkpoint, not guessed in code.
- No new APVTS params, no UI (Phase 10), no template import/export file format (ESCOPO §10 preset
  work, Phase 11). apply_hardware_template is a single-writer setup call (Phase 10 UI atomicizes).
- TR-8S advanced/assignable per-instrument routing beyond the default GM layout is out of scope;
  v1 template = default GM map shared with TR-8.
- Block-granular CC timing (09-03 limitation) unchanged.
- Real-hardware clock-jitter MEASUREMENT (numbers) is observational at the checkpoint, not an
  automated gate (no TR-8 in CI) — deferred.
- TR-8S advanced/assignable per-instrument routing + TR-8S-only CC banks beyond the shared GM
  default layout — deferred (Phase 10/11).
- Template import/export file format (load/save custom mappings) — deferred (ESCOPO §10, Phase 11).
