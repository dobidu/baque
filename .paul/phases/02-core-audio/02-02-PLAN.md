<objective>
## Goal
Wire host transport (BPM, playhead position, isPlaying) into the processor, implement sample-accurate MIDI event dispatch from the MidiBuffer (dispatch at the correct sample offset within processBlock, not at block boundary), and verify Phase 2 DoD: pad fires sample without clicks, sample-accurate, RT-safe.

## Purpose
Sample-accurate scheduling is non-negotiable for a beat machine — even a 1-block delay (~10ms at 44.1kHz/512) would make BAQUE unusable for tight grooves. This plan closes the Phase 2 loop so Phase 3 (Sequencer) can build on a solid timing foundation.

## Output
- `src/audio/scheduler.h/.cpp` — processes MidiBuffer with sample-accurate offsets, dispatches to VoicePool at correct position
- `src/plugin_processor.h/.cpp` — host transport reading, scheduler integration
- `tests/test_transport.cpp` — sample-accurate timing tests
</objective>

  <name>Task 1: Sample-accurate MidiBuffer scheduler</name>
  <files>src/audio/scheduler.h, src/audio/scheduler.cpp</files>
  <action>
    **Scheduler** (`src/audio/scheduler.h/.cpp`):
    - Single method: `void process(const juce::MidiBuffer& midi, VoicePool& pool, const float* sample_data, int num_samples, int block_size)`
    - Iterates `juce::MidiBuffer::Iterator` (or range-for with structured binding in JUCE 8)
    - For each MidiMessage at sample offset `pos`:
      - Note-on (any channel, any note): call `pool.trigger_at(pos, sample_data, num_samples, velocity_gain)` — triggers the voice but marks it to start at frame `pos` within the current block
    - `trigger_at(int start_offset, ...)` adds a field `start_offset_` to SampleVoice (default 0, set by trigger_at)
    - In `VoicePool::process_all(float* out_left, float* out_right, int num_frames)`: each active voice checks `if (current_frame < start_offset_) skip; else process` — this gives sample-accurate dispatch within the block
    - Note-off: mark voice inactive gracefully (short fade-out over 32 samples before setting inactive)
    ESCOPO §8 conventions. No allocation in any hot path.
  </action>

    **TransportState struct** (in plugin_processor.h, small POD):
    ```cpp
    struct TransportState {
        bool is_playing = false;
        double bpm = 120.0;
        double ppq_position = 0.0;  // position in quarter notes
        double sample_rate = 44100.0;
    };
    ```

    **In processBlock:**
    - Read `getPlayHead()` → `juce::Optional<juce::AudioPlayHead::PositionInfo> pos = getPlayHead()->getPosition()`
    - If valid: update `transport_` member (TransportState) from pos — extract bpm, ppq_position, isPlaying
    - If no playhead (unit tests): transport_ keeps last values
    - Replace direct MidiBuffer iteration in processBlock with `scheduler_.process(midiMessages, voice_pool_, sample_data_.data(), (int)sample_data_.size(), buffer.getNumSamples())`
    - After scheduler, call `voice_pool_.process_all(...)` then apply gain smoother

    This wiring replaces the naive block-boundary dispatch from 02-01 with the sample-accurate scheduler.
    TransportState is read-only in audio thread (written by getPlayHead which is audio-thread-safe in JUCE).
