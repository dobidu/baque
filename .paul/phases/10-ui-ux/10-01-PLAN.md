<objective>
## Goal
Make every engine struct the UI must touch safely writable from the message thread, and every value the UI must display safely readable — BEFORE any UI component exists. Retires the single-writer debt flagged in Phases 4/8/9: `perf_state_`, `lane_routing_`, `clock_master_`, `cc_out_`, pad params, live pattern/p-lock edits, `apply_hardware_template`.

## Purpose
Phase 10 carries an explicit entry constraint (STATE.md, audits 08-04 SR1 / 09-03 M1 / 09-04 SR1): these structs are single-writer, audio-thread-read; a UI writer without a command queue is a silent data race. Plans 10-02..10-07 all bind UI to engine state — this plan is the foundation they queue through.

## Output
- `src/audio/ui_command_queue.h` — POD `UiCommand` + SPSC `UiCommandQueue` (juce::AbstractFifo, fixed capacity, zero-alloc after construction)
- `src/audio/ui_state_snapshot.h` — `UiStateSnapshot`: per-field atomics published by the audio thread each block
- `BaqueProcessor`: `push_ui_command()` (message thread) + drain at top of `processBlock` + snapshot publish at end
- Tests: queue semantics + end-to-end command→engine-state integration
</objective>

<acceptance_criteria>

## AC-1: Command queue is RT-safe SPSC
```gherkin
Given a UiCommandQueue constructed with fixed capacity
When the message thread pushes N commands and the audio thread drains
Then commands arrive in FIFO order, drain performs no allocation/locks/IO,
  and push on a full queue returns false without blocking or overwriting
```

## AC-2: Every targeted struct is mutable via command, applied at block top
```gherkin
Given a running processBlock loop
When the UI pushes set_mute / set_solo / set_fill / set_lane_mode / set_lane_channel /
  set_clock_master / set_cc_out_* / set_pad_param / set_step / set_plock commands
Then the corresponding engine struct reflects the value before Sequencer::generate()
  runs in that same block (drain precedes generate), and audible/MIDI output proves it
```

## AC-3: Hardware template apply works through the queue with 09-04 semantics
```gherkin
Given an active pattern with user trigs and a tr8_template() apply command
When the audio thread drains the command
Then lane_routing_ + cc_out_ match the template, pattern notes for mapped lanes are
  rewritten, and existing trig activations/velocities/p-locks are preserved (SR1 09-04)
```

## AC-4: Snapshot readable from UI without torn single fields
```gherkin
Given processBlock running with the sequencer playing
When the message thread reads UiStateSnapshot at any moment
Then current step, transport playing flag, per-lane last-trigger velocity, and master
  peak L/R are each individually coherent (per-field atomic), updated every block
```

## AC-5: State save/load is race-free against audio-thread command application <!-- audit-added M1 -->
```gherkin
Given commands pushed to the queue (drained or not yet drained)
When the host calls getStateInformation or setStateInformation
Then audio processing is suspended for the bracket, pending commands are drained on the
  message thread BEFORE the state is read/written, and the saved state reflects every
  command pushed before the call — no concurrent audio-thread access to the structs
```

  <action>
    Create `UiCommandType : uint8_t` enum covering: set_fill, set_mute, set_solo,
    set_lane_mode, set_lane_channel, set_clock_master, set_cc_out_enabled,
    set_cc_out_channel, set_cc_slot (param index + cc number + enabled),
    set_pad_gain, set_pad_pan, set_pad_pitch_semitones, set_pad_pitch_cents,
    set_pad_reverse, set_pad_choke_group, set_pad_play_mode, set_pad_adsr_attack,

    clear_plock, apply_template (template id: 0=TR-8, 1=TR-8S).

    `UiCommand` = POD tagged union: `{ UiCommandType type; int32_t a; int32_t b; int32_t c; float f; }`.
    Static_assert trivially_copyable.

    `UiCommandQueue`: fixed `std::array<UiCommand, 256>` + `juce::AbstractFifo`.
    `bool push(const UiCommand&)` (message thread; false when full, never blocks).
    `template <typename Fn> void drain(Fn&& apply)` (audio thread; pops all pending,
    calls apply per command; no allocation, no locks — AbstractFifo is the established
    lock-free pattern, ESCOPO architecture note).
    Comment block (Portuguese, per convention) stating the SPSC contract: exactly one
    producer (message thread), one consumer (audio thread).
    <!-- audit-added SR2 --> Enforce the producer contract in debug builds: push() opens with
    `jassert(juce::MessageManager::existsAndIsCurrentThread());` — a stray worker-thread
    producer breaks SPSC silently in release; the assert catches it in development.
    Exception: drain may run on the message thread ONLY inside a suspendProcessing bracket
    (state save/load path, Task 2) — document this role-transfer rule in the same comment.

    Tests (ASCII names only — Windows ctest constraint):
    - QC1 FIFO order preserved across wrap-around (push/drain > capacity total)
    - QC2 push on full queue returns false, queue contents intact
    - QC3 drain on empty queue calls apply zero times
    - QC4 interleaved push/drain keeps order (single-threaded simulation of SPSC)
    Register test_ui_command_queue.cpp in tests/CMakeLists.txt add_executable list.
  </action>
  <verify>cmake --build build && ctest --test-dir build -R "QC" passes; full suite still 209+ green</verify>
